## 🐛 问题根源

### 问题 1: 文件存储 vs 云部署
- **本地开发**: 二维码保存为文件 `/qrcodes/user_xxx.png`
- **Railway 部署**: 文件存储在临时容器，重启后丢失
- **结果**: 数据库有路径，实际文件不存在 → 404

### 问题 2: 数据库字段长度
- **原字段类型**: `VARCHAR(255)`
- **Base64 编码长度**: ~2700 字符
- **结果**: `value too long for type character varying(255)`

---

## ✅ 最终解决方案

### 方案：Base64 编码存储

**优点**:
- ✅ 不需要文件存储
- ✅ 数据跟随数据库，不会丢失
- ✅ 前端可以直接使用 `data:image/png;base64,...`
- ✅ 适合云部署环境

## 🔧 修复内容

### 1. 后端修复

#### 修改文件
`backend/src/controllers/authController.js`

#### 修改内容

**生成二维码函数**:
```javascript
// 生成用户二维码（返回 base64 编码）
async function generateQRCodeBase64(user) {
  const baseUrl = process.env.BASE_URL || 'https://distribution-system-production.up.railway.app';
  const registerUrl = `${baseUrl}/register?ref=${user.referralCode}`;
  
  // 生成 base64 编码的 PNG
  const qrCodeDataUrl = await QRCode.toDataURL(registerUrl, {
    width: 300,
    margin: 2
  });
  
  return qrCodeDataUrl; // data:image/png;base64,...
}
```

**获取用户信息**:
```javascript
exports.getCurrentUser = async (req, res) => {
  const user = await User.findByPk(req.user.id);
  
  // 如果没有二维码或不是 base64 格式，重新生成
  if (!user.qrCodeUrl || !user.qrCodeUrl.startsWith('data:')) {
    const qrCodeBase64 = await generateQRCodeBase64(user);
    user.qrCodeUrl = qrCodeBase64;
    await user.save();
  }
  
  res.json({ code: 200, data: { user } });
};
```

## 🎯 技术要点

### Base64 编码优势
1. **无需文件存储**: 数据直接存在数据库
2. **云友好**: 适合无状态云部署
3. **前端友好**: 可以直接作为图片 src
4. **备份简单**: 数据库备份即包含二维码

### 性能考虑
- **存储空间**: 每个二维码约 2.7KB
- **传输开销**: 可接受（单个用户只传输一次）
- **缓存策略**: 前端可以缓存用户信息
